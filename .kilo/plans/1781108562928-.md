# Plan: Port pangolin and proton-pass-cli sysexts to the sysext-bakery model

## Context

Two CLI sysexts live today on the unmerged `starlit-sysexts` branch of the `starlit-os/dakota` fork as **BuildStream `.bst` elements** that produce directory-form sysexts, plus matching `just` recipes, `docs/sysexts/*.md` pages, and a `.github/workflows/sysext-publish.yml` that pipes artifacts to Cloudflare R2.

| Sysext | Element | Source |
|---|---|---|
| pangolin | `elements/sysext/pangolin.bst` (compose of `pangolin-cli.bst` + `pangolin-metadata.bst`) | `github_files:fosrl/cli@0.9.0` linux amd64/arm64 |
| proton-pass-cli | `elements/sysext/proton-pass-cli.bst` (compose of `…-cli.bst` + `…-metadata.bst`) | `github_files:protonpass/pass-cli@2.1.2` linux x86_64/aarch64 |

Both are `kind: manual` `.bst` elements that just `install -Dm755` an upstream prebuilt binary into `%{install-root}%{bindir}` and drop an `extension-release.<name>` into `%{indep-libdir}/extension-release.d/`. Neither is wired into the Dakota OCI image — they are host-side sysexts only.

The destination is a **`sysext-bakery`-style** repo, modelled on `flatcar/sysext-bakery` (Apache-2.0, Shell recipes, GitHub Actions publishing, release-based consumption via `systemd-sysupdate` + `SHA256SUMS` index, per-extension and global metadata releases). Target org for the new repo: `starlit-os` (the user already maintains a fork there; no `projectbluefin/sysext-bakery` exists).

**Out of scope** for this plan: the `starlit-cli` 12-tool bundle, `starlit-desktop`, the Cloudflare R2 publisher, the `justfiles/sysext*.just` workflow, and any docs in `docs/sysexts/`. These stay on `starlit-sysexts` (or get ported later in a separate effort).

---

## Why the bakery model

- Drops the BuildStream dependency for two trivial single-binary sysexts that don't need a junction, a stack, or a source build — `.bst` machinery is overkill and ties consumption to the dakota build pipeline.
- Produces `.raw` squashfs images (not directory form) that can be consumed by any systemd-sysext-capable host via `/var/lib/extensions/<name>.raw` or a sysupdate transfer.
- Pinned SHA256SUMS + per-extension releases + global metadata release give us a git-ops release story identical to the Flatcar bakery.
- All recipes are auditable Shell, with a self-contained GitHub Actions workflow for x86-64 and aarch64 (matches what `starlit-sysexts` already has in `sysext-publish.yml`).
- The bakery Apache-2.0 license is compatible with both `starlit-os/dakota` (MIT) and `projectbluefin/dakota` (MIT).

## Port — recipe contents

### `recipes/pangolin/create_pangolin_sysext.sh`

```sh
#!/usr/bin/env bash
# Adapted from the starlit-os/dakota BuildStream element
# elements/sysext/pangolin.bst (commit b9e27e3 / branch starlit-sysexts).
set -euo pipefail

VERSION="${VERSION:-0.9.0}"
ARCH="${ARCH:-$(uname -m)}"
case "$ARCH" in
  x86_64)  SUF=amd64   SHA=6056da7b3cea8a9ebc36c0ef9e003bf505f8957a38d11f445d070a0a56e46764 ;;
  aarch64) SUF=arm64   SHA=39427a82db6f77d5ba6e61c11e0b1c022aa93542efbf606a2723b5198ceb9f3e ;;
  *) echo "unsupported arch: $ARCH" >&2; exit 1 ;;
esac

URL="https://github.com/fosrl/cli/releases/download/${VERSION}/pangolin-cli_linux_${SUF}"
DEST="${1:-pangolin.raw}"
ROOT="$(mktemp -d)"
trap 'rm -rf "$ROOT"' EXIT

curl -fsSL -o "$ROOT/pangolin" "$URL"
echo "${SHA}  $ROOT/pangolin" | sha256sum -c -

mkdir -p "$ROOT/usr/bin" "$ROOT/usr/lib/extension-release.d"
install -m 0755 "$ROOT/pangolin" "$ROOT/usr/bin/pangolin"

ARCH_REL="x86-64"; [ "$ARCH" = "aarch64" ] && ARCH_REL="arm64"
cat >"$ROOT/usr/lib/extension-release.d/extension-release.pangolin" <<EOF
ID=bluefin-dakota
ARCHITECTURE=${ARCH_REL}
VERSION_ID=0
EOF

mksquashfs "$ROOT" "$DEST" -comp xz -noappend -all-root
```

## Port — CI

### `.github/workflows/bake.yml`

- Trigger: `push` to `main` (paths under `recipes/`, `base/`, `tools/`, `.github/workflows/bake.yml`) and `workflow_dispatch`.
- Matrix: `arch: [x86-64, arm64]` × `recipe: [pangolin, proton-pass-cli]`.
- Steps per matrix cell:
  1. Checkout.
  2. `sudo apt-get install -y squashfs-tools mksquashfs` (and `qemu-user-static` + `binfmt` on the aarch64 leg for `uname -m`).
  3. `chmod +x recipes/<r>/create_<r>_sysext.sh`.
  4. Run script → `out/<recipe>-<sha>-<arch>.raw`.
  5. `sha256sum` → append to `out/SHA256SUMS`.
- Aggregate job: collect artifacts, upload to a per-extension `release` named `<recipe>-<short-sha>` containing the two `.raw` files and `SHA256SUMS`. Also create/update a per-recipe `latest` symlink release, and a global `sysexts-YYYYMMDD` metadata release that bundles all `SHA256SUMS` files.
- Anchored cosign verification per `AGENTS.md`: `--certificate-identity-regexp` pinned to `^https://github.com/starlit-os/sysext-bakery/.github/workflows/bake\.yml@refs/heads/(main|gh-readonly-queue/main/.+)$`.
- Runners: `blacksmith-8vcpu-ubuntu-2404` (same as the existing dakota `sysext-publish.yml`); for arm64, use a self-hosted aarch64 runner or a QEMU-emulated leg until aarch64 hardware runners are available — match the note already in `sysext-publish.yml`.
- Permissions: `contents: read`, `actions: read`; uses `GH_TOKEN` with `contents: write` only for the release job.

### `tools/bake_all.sh` (local dev helper)

Local-build entry point so maintainers don't have to push to test a recipe change. Mirrors the Flatcar bakery's `tools/bake_all.sh`. Iterates `recipes/*/create_*_sysext.sh`, builds for the local arch, prints a SHA256SUMS table.

### `base/create_sysext.sh`

Vendored copy of the Flatcar bakery's helper for any common post-install steps (currently unused by the two single-binary recipes, but included so future recipes can share the scaffolding). Add a `LICENSE` attribution block referencing `flatcar/sysext-bakery` commit and the Apache-2.0 origin.

- "Adding a new recipe" — five steps: create `recipes/<name>/`, add `create_<name>_sysext.sh` from `tools/template.sh`, add `sysupdate.<name>.conf`, add a row to `recipes/README.md`, open a PR.
- Versioning rules: pin upstream version in the script, bump SHA in the same commit, never edit published tags.
- Build locally: `./tools/bake_all.sh`.

### `docs/howto-consume.md`

Per-sysext Butane/Ignition snippet + manual install steps. Copied and adapted from the dakota-side `docs/sysexts/pangolin.md` and `docs/sysexts/proton-pass-cli.md`. The `sudo install -d /var/lib/extensions` + `cp -a` + `systemctl restart systemd-sysext.service` flow becomes `curl … | sudo install -m 0644 -t /var/lib/extensions …raw` + same restart.

## Execution steps

1. **Pre-flight (read-only)**
   - Confirm `starlit-os/sysext-bakery` does not yet exist (it shouldn't — `projectbluefin` org has no such repo, and `starlit-os` is the user's fork org). If it does, switch to "extend" mode and merge into the existing tree.
   - Read `flatcar/sysext-bakery` upstream `base/create_sysext.sh`, `tools/bake_all.sh`, `.github/workflows/bake.yml` to confirm shape matches (no need to re-derive).

2. **Create the repo**
   - On GitHub: `starlit-os/sysext-bakery`, Apache-2.0, `main` default branch, no template.
   - Locally: `gh repo create starlit-os/sysext-bakery --public --license Apache-2.0 --description "systemd-sysext recipes for bluefin-family hosts"`.

3. **Scaffold the repo**
   - Write `README.md`, `LICENSE`, `.gitignore` (ignoring `out/`, `*.raw`, `SHA256SUMS`).
   - Vendor `base/create_sysext.sh` from Flatcar with Apache-2.0 attribution.
   - Add `.github/workflows/bake.yml`, `.github/dependabot.yml`, `tools/bake_all.sh`, `tools/template.sh`, `docs/developer.md`, `docs/howto-consume.md`.

4. **Port the two recipes**
   - Create `recipes/pangolin/` and `recipes/proton-pass-cli/` per the script + sysupdate.conf + README above.
   - Add rows to `recipes/README.md`.

5. **Verify locally before push**
   - `sudo apt-get install -y squashfs-tools`
   - `./tools/bake_all.sh` — should produce `out/pangolin-x86-64.raw` and `out/proton-pass-cli-x86-64.raw` and a `SHA256SUMS`.
   - Mount each `.raw` with `unsquashfs -d /tmp/check recipes/.../*.raw` and confirm `usr/bin/<bin>`, `usr/lib/extension-release.d/extension-release.<name>`, and that `ID=bluefin-dakota` is set.
   - If a Dakota VM is reachable: `systemd-sysext merge` after dropping the raw into `/var/lib/extensions/` and confirm `<bin> --help` runs. AGENTS.md says no WIP PRs, but the local verification is required before opening the PR.

6. **First publish**
   - Push to `main`, let `bake.yml` run the matrix, confirm both x86-64 and arm64 `.raw` files land in the `<recipe>-<sha>` release with a `SHA256SUMS`.
   - Promote to `latest` symlink release.

7. **Cross-link from dakota**
   - Open a draft PR on `starlit-os/dakota` `starlit-sysexts` branch that replaces `docs/sysexts/pangolin.md` and `docs/sysexts/proton-pass-cli.md` with one-paragraph pointers to the bakery repo + release URL. Keep the `.bst` elements in place until the bakery has shipped a tagged release. (This is a draft only — not for merge until the bakery side is green.)

8. **Skill writeback (per AGENTS.md self-improvement loop)**
   - Add `docs/skills/sysext-bakery.md` to `starlit-os/dakota` covering: when to prefer the bakery model over a `.bst` element, the Apache-2.0 attribution requirement when vendoring from `flatcar/sysext-bakery`, and the `ID=bluefin-dakota` portability gate.
   - Register it in `docs/skills/README.md` routing table in the same PR.
